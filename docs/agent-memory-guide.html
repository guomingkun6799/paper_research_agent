2 记忆的三层分类体系

受认知科学启发，Agent 社区形成了 HOT / WARM / COLD 三层记忆共识。

🔥

HOT Memory（工作记忆）

当前正在做的事。直接注入 LLM 上下文窗口，每次调用都可访问。必须精简，控制在 ~2000 字以内。

最近经历过的具体事件。按语义相关性检索，按需注入上下文。覆盖范围：最近几天到几周。

3 七大主流记忆方案详解

从最简单的 Prompt 注入到最复杂的时态知识图谱，逐一剖析。

M

MemGPT / Letta

⭐ 14k+ Python 2023年10月 UC Berkeley
虚拟上下文管理 自编辑记忆 OS 级抽象 分页加载
核心思想：把 LLM 上下文窗口当"内存"，外存当"硬盘"，自己管理换页。 ┌───────────────────────────────────────────────────┐ │ LLM Context Window（"主存"，有限） │ │ ┌──────────┬──────────┬───────────────────────┐ │ │ │ System │ Core │ Recent Messages │ │ │ │ Prompt │ Memory │ (最近对话) │ │ │ └──────────┴──────────┴───────────────────────┘ │ │ ▲ │ │ │ │ 全量注入 │ 按需换页（send_message） │ │ │ ▼ │ │ ┌──────────┐ ┌──────────────┐ │ │ │ inner │ │ recall_memory │ ← Agent 自主决定什么时候加载什么 │ │ │ dialogue │ │ search_memory │ │ │ │ buffer │ │ insert_memory │ │ │ └──────────┘ └──────────────┘ │ │ │ │ ├──────────────────────────┼──────────────────────────┤ │ External Store（"硬盘"，无限） │ │ ┌──────────────────────────────────────────────┐ │ │ │ recall_storage (长期对话历史) │ │ │ │ archival_storage (归档记忆，向量检索) │ │ │ └──────────────────────────────────────────────┘ │ └───────────────────────────────────────────────────┘

MemGPT 的核心创新是让 Agent 像操作系统管理虚拟内存一样管理自己的上下文。Agent 有 inner_monologue（内心独白）和对外输出两层，通过 send_message、recall_memory、insert_memory 等内部函数自主决定何时从外存加载什么内容到上下文。

✅ 优势

  • Agent 自主管理记忆，无需人工干预
  • 理论上支持无限长度对话
  • 开源社区活跃，Letta 持续维护
  • 支持多种后端存储（Postgres / Chroma / Redis）

❌ 劣势

  • 记忆检索质量依赖 Agent 的 prompt 设计
  • 频繁的工具调用增加延迟和 token 成本
  • 缺乏时序推理能力（不知道"什么时候学的"）
  • 多 Agent 协作时记忆隔离不清晰
0

Mem0

⭐ 25k+ Python 2024年3月 Mem0 Inc
自动记忆提取 去重合并 用户/会话/Agent 三级 图记忆
核心思想：从对话中自动提取记忆片段，去重合并，形成结构化记忆。 对话/交互 │ ▼ ┌──────────┐ ┌─────────────┐ ┌──────────────┐ │ add() │────▶│ Memory Extractor │────▶│ 提取的记忆片段 │ │ 传入对话 │ │ (LLM 自动提取) │ │ "用户偏好Go" │ └──────────┘ └─────────────┘ └──────┬───────┘ │ ▼ ┌─────────────┐ │ 去重 + 合并 │ 相似记忆自动合并更新 └──────┬──────┘ │ ▼ ┌─────────────────────────────────────────────────┐ │ Mem0 Memory Store │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ 用户记忆 │ │ 会话记忆 │ │ Agent记忆 │ │ │ │ user_id │ │ session │ │ agent_id │ │ │ └──────────┘ └──────────┘ └──────────┘ │ └─────────────────────────────────────────────────┘

Mem0 的最大特点是开箱即用的自动记忆提取。你只需要调用 memory.add(data, user_id="xxx")，它就自动用 LLM 从对话中提取关键信息，存入向量库。检索时 memory.search(query, user_id="xxx") 返回相关记忆。2024年底还新增了 Graph Memory（图记忆）功能。

# Mem0 最简用法 — 3 行代码 from mem0 import Memory m = Memory() # 自动从对话提取记忆 m.add("我在做一个科研Agent，使用Go语言和Eino框架", user_id="mingkun") # 语义检索 result = m.search("用什么语言开发Agent", user_id="mingkun") # → ["用户在用Go和Eino框架开发科研Agent"]

✅ 优势

  • API 极简，3 行代码接入
  • 自动提取 + 去重 + 合并，零配置
  • 用户/会话/Agent 三级隔离
  • 支持 Qdrant / Chroma / Postgres 等多种后端

❌ 劣势

  • 记忆提取依赖额外 LLM 调用（成本增加）
  • 缺乏时序信息（不记录"什么时候"）
  • Graph Memory 功能较新，成熟度不足
  • 不适合需要推理的复杂知识关联场景
H

Hermes Agent — 有界双文件记忆

⭐ 46k+ Python 2024年 Nous Research
有界记忆 ≤2200 字 冻结快照 零额外成本 压缩摘要
核心思想：严格限制记忆大小（≤2200 字），不依赖向量库，用"写 + 压缩 + 快照"管理记忆生命周期。 ┌───────────────────────────────────────────────┐ │ MEMORY.md（≤ 2200 字，HOT Memory） │ │ ┌───────────────────────────────────────────┐│ │ │ # 2026-04-17 ││ │ │ - GNN 节点分类实验：GCN baseline 81.5% ││ │ │ - GraphSAGE 两层 84.2%，提升 2.7% ││ │ │ - 下一步：尝试 GAT 注意力机制 ││ │ └───────────────────────────────────────────┘│ └───────────────────────────┬───────────────────┘ │ 当字数超 2200 ▼ ┌───────────────────────────────────────────────┐ │ 压缩机制 │ │ 1. 保留最近 N 条 + 最重要结论 │ │ 2. 旧内容 LLM 摘要压缩 │ │ 3. 生成 snapshot_20260417.md 冻结快照 │ └───────────────────────────┬───────────────────┘ │ ▼ ┌───────────────────────────────────────────────┐ │ snapshots/（归档，不注入上下文，按需检索） │ │ ├── snapshot_20260410.md │ │ ├── snapshot_20260415.md │ │ └── snapshot_20260417.md │ └───────────────────────────────────────────────┘

Hermes 的哲学是"约束即自由"——严格限制记忆在 2200 字以内，迫使 Agent 只保留真正重要的信息。这避免了"记忆膨胀"问题（上下文塞满无用信息反而降低推理质量）。超限时自动摘要压缩 + 冻结快照，兼顾成本和质量。

✅ 优势

  • 零额外基础设施（纯文件，无需数据库）
  • 零额外 LLM 调用成本（压缩时才用）
  • 记忆质量高（有限空间倒逼精选）
  • 可解释性强（直接读文件就能看记忆内容）

❌ 劣势

  • 无法存储大量历史细节
  • 语义检索能力弱（纯文本匹配）
  • 多 Agent 共享记忆困难
  • 快照检索需要额外机制
A

AI Scientist v2 — 实验树 + 挂起记忆

⭐ 15k+ Python 2025年 Sakana AI
BFTS 树搜索 实验节点记忆 自动剪枝 SOTA 追踪
核心思想：记忆附着在实验树节点上，搜索过程即记忆积累过程。 [Root: Idea] / | \ [Idea A] [Idea B] [Idea C] ← 每个节点保存： / \ | - 假设描述 [Exp A1] [A2] [Exp B1] - 代码快照 │ │ │ - 实验结果 ▼ ▼ ▼ - 错误日志 Success! Failed 82.3% - 评分 记忆传递规则： - 父节点结论 → 子节点 prompt（不重复犯相同错误） - 兄弟节点结果 → 评分参考（避免无用探索） - 失败节点 → 错误模式库（3 次失败自动剪枝）

✅ 优势

  • 记忆和实验深度绑定，天然结构化
  • BFTS 并行探索效率极高
  • 失败记忆自动传播，避免重蹈覆辙
  • 搜索树本身就是可视化的记忆呈现

❌ 劣势

  • 只适合实验探索场景，非通用
  • 跨研究课题的记忆不持久
  • 树深度过大时记忆检索效率下降
Z

Zep / Graphiti — 时态知识图谱记忆

⭐ 3.5k+ Python + TypeScript 2024年 Zep AI
时态知识图谱 实体关系提取 时间推理 LongMemEval 第一
核心思想：记忆不只是文本片段，而是带时间戳的实体-关系图谱，支持时序推理。 输入："[2026-04-15] GraphSAGE 在 Cora 上达到 83.5%" │ ▼ ┌──────────────────────────────────────────────────┐ │ Graphiti 知识提取管道 │ │ │ │ 实体提取 ──▶ 关系提取 ──▶ 时间戳绑定 ──▶ 图谱更新 │ │ │ │ GraphSAGE ──achieved──▶ 83.5% ──on──▶ Cora │ │ │ │ │ │ │ └──[created: 2024-01-01] │ │ │ └──[date: 2026-04-15] │ │ │ │ 时间推理能力： │ │ Q: "Cora 上最好的模型是什么？" │ │ → GraphSAGE (83.5%, 2026-04-15) │ │ → 知道 GAT 曾在 2026-03 达到 82.1% │ │ → 知道 GCN 在 2025-12 只有 81.5% │ │ → 结论：GraphSAGE 当前 SOTA │ └──────────────────────────────────────────────────┘

✅ 优势

  • 支持时序推理（"什么时候"、"当前最好"）
  • 实体关系天然适合科研知识（方法→结果→数据集）
  • LongMemEval 基准测试第一名
  • 自动从非结构化文本提取结构化知识

❌ 劣势

  • 依赖 Neo4j，基础设施较重
  • 实体提取需要 LLM 调用，成本较高
  • 图谱过大时查询性能需优化
  • 生态较新，社区资源不如 Mem0 丰富
G

GraphRAG (Microsoft Research)

⭐ 20k+ Python 2024年7月 Microsoft Research
社区检测 层次化摘要 全局理解 Leiden 算法
核心思想：用 LLM 从文档中提取实体关系图谱 + 社区层次摘要，实现全局理解。 文档集合 │ ▼ ┌──────────┐ ┌───────────┐ ┌──────────────┐ │ Text Chunk │───▶│ Entity/Relation │───▶│ Knowledge │ │ 分块 │ │ 提取 │ │ Graph │ └──────────┘ └───────────┘ └──────┬───────┘ │ ▼ ┌──────────────┐ │ Leiden 社区检测 │ 自动发现实体聚类 └──────┬───────┘ │ ┌────────────┼────────────┐ ▼ ▼ ▼ 社区A摘要 社区B摘要 社区C摘要 │ │ │ └────────────┼────────────┘ ▼ 全局摘要 (Global Summary)

✅ 优势

  • 超越普通 RAG 的全局理解能力
  • 社区检测自动发现知识聚类
  • 层次化摘要兼顾局部和全局查询
  • 微软官方维护，文档完善

❌ 劣势

  • 索引阶段成本极高（大量 LLM 调用）
  • 不支持时态信息（不知道"何时"）
  • 更新图谱需要重新索引（增量更新差）
  • 适合静态文档，不适合动态 Agent 记忆
R

滑动窗口 + RAG（基线方案）

— Any 经典方案
简单直接 滑动窗口 向量检索 低成本
核心思想：保留最近 N 轮对话 + 向量检索相关历史。 对话历史：[msg1, msg2, msg3, ..., msgN, msgN+1, msgN+2(当前)] ◄──── 滑动窗口 ────► 每次调用时： 1. 取最近 K 轮对话（窗口内） 2. 用当前问题查向量库 Top-5 历史片段 3. 拼接：System Prompt + 窗口对话 + 检索片段 + 当前问题 4. 发给 LLM

✅ 优势

  • 实现最简单，任何框架都能快速集成
  • 零额外 LLM 成本（无提取/摘要步骤）
  • 延迟最低（只需一次向量检索）
  • 成熟稳定，生产验证充分

❌ 劣势

  • 窗口外的信息永久丢失
  • 无结构化知识，无法推理
  • 向量检索可能召回无关内容
  • 完全不具备时序理解能力

6 科研 Agent 推荐方案

针对自主科研场景的最优记忆架构组合。

🎯 推荐方案：三层混合记忆架构

综合七种方案的优劣，科研 Agent 最适合 HOT(有界文件) + WARM(向量库) + COLD(知识图谱) 的三层混合架构。HOT 层借鉴 Hermes 的约束哲学，WARM 层使用 Chroma/RAG，COLD 层使用 Graphiti 的时态知识图谱处理方法对比关系。

🔥 HOT Memory
CURRENT_STATE.md ≤ 2200 字
借鉴 Hermes：精简约束，零成本，每次调用注入
🌡️ WARM Memory
Chroma 向量数据库
RAG 检索历史实验记录、文献摘要，按需注入
🧊 COLD Memory
Graphiti 时态知识图谱
时序推理：追踪方法超越关系，回答"当前 SOTA 是什么"
🌳 实验记忆
BFTS 树节点（SQLite）
借鉴 AI Scientist：实验树即记忆，父子节点传递结论
完整记忆读写时序： Agent 收到任务: "继续 GNN 节点分类实验" │ ├─ 1. 读取 HOT: CURRENT_STATE.md │ → "上次 GraphSAGE 跑到 84.2%，下一步试 GAT" │ → 直接注入 System Prompt │ ├─ 2. 查询 WARM: Chroma 语义检索 │ → 检索到 GraphSAGE 的完整代码和实验配置 │ → 检索到相关论文笔记 │ → 注入 Context Window │ ├─ 3. 查询 COLD: Graphiti 知识图谱 │ → Q: "Cora 数据集上最好的 GNN 方法" │ → GCN(81.5%, 2025-12) → GraphSAGE(84.2%, 2026-04) → ... │ → 发现：GAT 在类似数据集上通常比 GraphSAGE 高 1-2% │ → 注入 Context Window │ ├─ 4. 生成 GAT 实验代码 → Docker 执行 │ ├─ 5. 实验完成 → 更新记忆 │ ├─ HOT: 更新 CURRENT_STATE.md（"GAT 达到 85.1%"） │ ├─ WARM: 存入 Chroma（实验记录向量） │ ├─ COLD: 更新 Graphiti（GAT→85.1%→Cora 实体） │ └─ 树节点: 记录到 BFTS 树 │ └─ 6. 检查 HOT 字数 → 超 2200 字 → 压缩 + 冻结快照

为什么不选单一方案？科研 Agent 的记忆需求是多维的：需要即时上下文（HOT）、历史经验检索（WARM）、时序知识推理（COLD）、实验结构管理（BFTS 树）。没有任何单一方案能同时满足这四个维度，混合架构是必然选择。
